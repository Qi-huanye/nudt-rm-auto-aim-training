# 第 10 课：入门改造任务

## 本课目标

- 把前面课程中的阅读理解转化成工程实践。
- 通过小范围、可验证的任务熟悉代码修改流程。
- 学会为每个改动写清楚原因、验证方法和结果。

## 前置知识

- 已完成第 0 到第 9 课。
- 能解释主数据流、接口消息、启动系统、相机、识别、跟踪、弹道和串口。

## 课堂内容

### 1. 任务选择原则

入门改造任务必须满足：

- 范围小。
- 修改点明确。
- 能独立验证。
- 不依赖大量硬件条件。
- 不混入无关重构。

### 2. 推荐任务池

任务 A：给关键参数补中文说明。

- 目标文件：`node_params.yaml`、`launch_params.yaml`。
- 重点：只补说明，不改参数值。
- 验证：配置文件语义更清楚，launch 行为不变。

任务 B：修复 YOLO 模型硬编码路径。

- 目标模块：`armor_detector`。
- 重点：把绝对路径改为 ROS 参数或 package share 路径。
- 验证：缺模型时报错清晰，有模型时能加载指定路径。

任务 C：清理 launch 中失效包名。

- 目标模块：`vision_bringup`。
- 重点：确认实际包名是 `armor_detector`，不要引用不存在的 `armor_detector_rensy`。
- 验证：launch 不再因为找不到包失败。

任务 D：写假 `Target` 发布节点或命令方案。

- 目标模块：`solve_angle`。
- 重点：无需真实识别链路即可测试 `/gimbal_command`。
- 验证：固定输入能产生稳定输出。

任务 E：写模拟串口工具设计。

- 目标模块：`jlcv_serial_driver`。
- 重点：模拟下位机收发包。
- 验证：能触发 `/game_status`、`/joint_states` 或接收 `/gimbal_command`。

任务 F：给 tracker 增加状态 debug 输出。

- 目标模块：`armor_processor`。
- 重点：发布或打印 tracker 状态，帮助理解 `LOST/DETECTING/TRACKING/TEMP_LOST`。
- 验证：状态变化可观察。

任务 G：用 rosbag 对比 EKF 参数。

- 目标模块：`armor_processor`。
- 重点：只改参数，不改算法。
- 验证：同一段数据下比较目标轨迹变化。

### 3. 改造报告模板

每个任务完成后写一份简短报告：

```text
任务名称：
修改原因：
涉及文件：
修改内容：
验证命令：
验证结果：
风险和后续：
```

### 4. 验证标准

每个改动都要有证据：

- 构建是否通过。
- 对应节点是否能启动。
- topic 是否有预期输出。
- 参数是否生效。
- 错误路径是否更清晰。

如果因为缺硬件无法完整验证，需要写明替代验证方式和未覆盖风险。

## 实践任务

至少选择 2 个任务完成详细设计，其中至少 1 个应可以实际实现。

推荐组合：

- 任务 A + 任务 C：适合刚开始熟悉配置和 launch。
- 任务 B + 任务 D：适合开始动 C++ 节点和测试。
- 任务 E + 任务 F：适合准备联调和 debug 能力。

每个任务都需要写：

- 目标。
- 修改文件。
- 修改步骤。
- 验证步骤。
- 回滚方案。

## 检查问题

- 这个任务的最小修改范围是什么？
- 如何证明改动生效？
- 如果验证失败，如何判断是改动问题还是环境问题？
- 哪些文件不应该被这个任务顺手修改？
- 这个改动会影响哪些节点或 topic？

## 课程完成条件

完成本课需要同时满足：

- 至少选择 2 个入门任务并写出详细设计。
- 至少完成 1 个可验证的小改造，或在缺硬件时完成等价的静态/模拟验证。
- 每个任务都有明确的涉及文件、修改步骤、验证步骤和风险说明。
- 能说明改动的影响范围，不把无关重构混入任务。
- 能提交一份改造报告，包含修改原因、验证命令和验证结果。
