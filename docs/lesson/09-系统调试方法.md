## 课堂内容

### 1. 总原则

排查顺序应从数据源头往后走：

```text
图像
  -> 相机内参
  -> 识别结果
  -> TF
  -> tracker
  -> 弹道
  -> 串口
  -> 下位机执行
```

不要看到“打不准”就直接改弹道参数。很多打不准来自识别、标定、TF 或下位机角度定义。

### 2. 没图像

检查：

- 相机 SDK。
- USB 权限。
- 相机序列号。
- 曝光、增益、帧率。
- `/image_raw` 是否存在。
- `/camera_info` 是否存在。

### 3. 有图像没识别

检查：

- `armor_detector` 是否启动。
- 模型路径是否存在。
- `enemy_color` 是否正确。
- debug 图像是否正常。
- 阈值和忽略类别。

### 4. 有识别没跟踪

检查：

- `/detector/armors` 是否有数据。
- `Armor.pose` 是否稳定。
- TF 是否连通。
- `target_frame` 是否正确。
- tracker 匹配阈值。
- tracker 状态。

### 5. 有目标没控制

检查：

- `/processor/target.tracking`。
- `solve_angle` 是否启动。
- TF 是否能查到 `pitch_link`。
- `/gimbal_command` 是否发布。

### 6. 有控制但下位机不动

检查：

- 串口设备名。
- 波特率。
- CRC。
- 包头。
- 下位机协议字段顺序。
- control_status 是否为预期值。

### 7. 打不准

检查：

- 图像识别是否稳定。
- 相机内参是否正确。
- 外参和 TF 是否正确。
- 目标预测是否合理。
- 弹速和延迟是否合理。
- offset 是否只用于小范围补偿。
- 下位机角度单位和符号是否一致。

## 实践任务

1. 写一份问题排查 checklist，按模块分层。
2. 写一份调参记录模板，至少包含：

- 日期
- 场景
- 问题现象
- 修改参数
- 修改前值
- 修改后值
- 验证方式
- 结果
- 是否保留

3. 选择一个假设问题，例如“有识别但没有 `/processor/target`”，写出排查路径。
4. 选择一个假设问题，例如“有 `/gimbal_command` 但车不动”，写出排查路径。

## 检查问题

- 为什么排查要从数据源头开始？
- “有图像没识别”和“有识别没跟踪”的分界 topic 是什么？
- “有目标没控制”和“有控制但下位机不动”的分界 topic 是什么？
- offset 能解决哪些问题，不能解决哪些问题？
- 为什么每次调参只改一个变量？

## 课程完成条件

完成本课需要同时满足：

- 能写出从图像到下位机的分层排查链路。
- 能为至少 4 类常见问题写出检查步骤。
- 能明确每类问题的分界 topic 或关键证据。
- 能写出调参记录模板。
- 能说明为什么不应该盲目同时修改多个参数。
